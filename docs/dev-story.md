# 🛠️ 开发故事：lark-retro 是怎么做出来的

> 从 v1.0 到 v2.0，一个纯 SKILL.md 的 Sprint 回顾工具背后的决策、踩坑和迭代。

---

## 为什么选择纯 SKILL.md，而不是写脚本？

最开始我也想过写一个 Shell 脚本，把 `lark-cli` 命令串起来跑。后来发现这条路走不通：

**脚本是死的，回顾是活的。**

每次回顾的数据量不同、关注点不同。有时候你想看消息，有时候你只想看日历和任务。脚本跑出来的结果是固定的，你要的是"理解数据后给出洞察"。这件事只有 AI 能做。

纯 SKILL.md 的好处是——AI Agent 会读你的指令，然后**根据实际拿到的数据动态决定怎么分析、怎么呈现**。日程多就重点分析会议模式，任务多就重点看完成率趋势，消息多就提取关键讨论。这不是预设的 if-else，是 AI 在理解数据后做的判断。

代价是什么？你得把指令写得足够精确。模糊的指令 = 不稳定的输出。所以 SKILL.md 写到了 591 行。

## 踩坑记录

### 坑 1：消息搜索全是噪声

第一次跑 `im +messages-search`，返回了 50 条消息。打开一看，全是社区群的闲聊、机器人通知、表情包回复。有价值的工作讨论不到 5 条。

如果把这些噪声喂给 AI，生成的报告会出现"本周在社区群里讨论了如何安装 lark-cli"这种毫无价值的总结。

**解决方案：4 层过滤漏斗**

```
原始消息 → 来源区分（群聊/单聊/Bot）
         → 类型排除（system/表情/红包/投票）
         → 关键词排除（机器人通知/广告/水群）
         → 质量保留（有实质内容的讨论）
```

这个漏斗不是一次设计出来的，是跑了十几次真实数据后逐步加上的。每次发现一类噪声就加一条规则。

后来 v2.0 加了 `im +chat-messages-list` 作为第二通道——如果用户指定了群聊，就直接拉这个群的消息列表，比搜索精准得多。

### 坑 2：日历里 38 个"AI切磋大会"

实测时发现：一周的日程返回了 52 条，其中 38 条是同一个活动的不同城市分会场（北京、上海、深圳、成都、杭州云谷、杭州东站、库尔勒、慈溪……）。

不做聚合的话，AI 会认为你这周参加了 38 个会议，然后得出"会议严重超载"的结论。

**解决方案：同名聚合规则**

> 去掉 summary 中的地名/数字后缀，若剩余部分相同则视为同一活动，合并为 1 条，标注实际参加的实例。

这个规则看起来简单，但"地名"的定义是模糊的。"杭州"是地名，"云谷"算不算？"库尔勒"呢？最后的策略是：不追求完美匹配，只要同一天同一时段、组织者相同、名称相似度 > 80%，就视为同一组。

### 坑 3：任务为空，要不要问用户？

很多人（包括我自己）不在飞书任务里管理待办，用的是 Jira、Linear 或者 Notion。

v1.4 的做法是：发现任务为空时，弹出来问用户"你是不是用其他工具管理任务？"。听起来很贴心，但实际体验很差——用户说"帮我做回顾"，期望一键完成，结果中间停下来问问题，工作流被打断了。

**v2.0 改成了非阻塞式**：任务为空就在报告的"数据质量说明"里标注"✅ 任务数据 ❌"，不打断流程。末尾补一句提示："如果您使用 Jira/Linear 等外部工具，可在下次回顾时告知。"

原则是：**自动化工作流里，能不问就不问。**

### 坑 4：`--as user` vs `--as bot`

lark-cli 有两种身份。刚开始我没在意，全用 `--as user`。后来发现：

- `im +messages-send`：用 user 身份发消息需要额外的 `im:message.send_as_user` scope，审批流程特别长
- `docs +create`：user 和 bot 都行，但 bot 创建的文档归属不同
- `calendar +agenda`：只有 user 能读自己的日程

**最终策略**：SKILL.md 里做了一个身份选择表，每个命令标注了推荐身份和回退方案。

### 坑 5：`--jq` 遇到 `null` 会静默失败

v2.0 加了 `--jq` 优化来减少 token 消耗。比如：

```bash
lark-cli task +get-my-tasks --jq '.data.items[] | {summary, due}'
```

跑得很好——直到任务列表为空。`items` 是 `null` 时，`--jq` 不会报错，直接返回空输出。AI Agent 看到空输出，不知道是"没有数据"还是"命令出错了"。

**解决方案**：在 SKILL.md 里加了边界提醒：

> "⚠️ 当 `items: null` 时 `--jq` 过滤器会返回空输出。先不带 `--jq` 确认数据非空，再使用过滤。"

### 坑 6：上期报告怎么找？

闭环追踪需要找到上一次的回顾报告。问题是：报告标题可能是"Sprint 回顾 W12"，也可能是"第 12 周回顾"，也可能是用户自己改过的标题。

`docs +search` 是模糊搜索，用"Sprint 回顾"去搜，可能搜出一堆不相关的文档。用太精确的关键词又可能搜不到。

**最终策略**：

1. 先用"Sprint 回顾 W{上期周数}"精确搜索
2. 搜不到就放宽为"回顾报告"
3. 搜到多个就按时间排序取最近的
4. 实在搜不到就跳过对比，标注"未找到上期报告"

v2.0 还加了 `drive +export` 可以把搜到的文档导出为 Markdown 做全文对比，但这个需要额外的 scope 授权，所以做成了可选功能。

## 版本演进

| 版本 | 核心变化 | 驱动力 |
|------|---------|--------|
| v1.0 | 基础回顾：日历 + 任务 → 报告 | 最小可用 |
| v1.4 | 消息搜索 + 噪声过滤 + 知识库归档 | 数据源不够 |
| v1.5 | 报告模板加 emoji + 视觉优化 | 输出太素 |
| v2.0 | 行动项闭环 + 周报模式 + `--jq` + 任务列表 | 从"能用"到"好用" |

每个版本的升级都不是提前规划好的，而是在实际使用中发现了新的痛点。比如 v2.0 的行动项闭环，是因为我连续做了三周回顾后发现：**每次都有行动项，但从来没人追踪过。** 这和不做回顾有什么区别？

## 一个设计原则：渐进增强

从 v1.0 开始我就坚持一个原则：**缺少任何一个授权，都不能导致整个 Skill 挂掉。**

很多工具的做法是：启动时检查所有权限，缺一个就报错退出。用户面对一大堆权限申请，还没开始用就放弃了。

lark-retro 的做法是分 4 层：

- 🟢 只有日历 + 文档 → 能做基础回顾
- 🔵 加上任务 → 能追踪完成率
- 🟣 加上消息搜索 → 能分析关键讨论
- 🟠 加上 Bot → 能群聊通知

每一层都独立可用，用户可以从最小权限开始体验，觉得好用了再逐步开启更多能力。
